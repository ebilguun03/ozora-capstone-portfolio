Capstone Reflection

Overall Capstone Reflection

Ozora pushed me to think beyond a single app screen and treat the project as a connected system. I ended up building a SwiftUI client, an Express backend, an admin dashboard, and a set of demo scripts that made the whole experience easier to explain, test, and revise. That shift from feature thinking to system thinking was one of the most important outcomes of the capstone.

What I Learned About Scope

Early in the project, the obvious temptation was to keep adding restaurant features. The more valuable lesson was learning how to protect the strongest story: a guest opening a session, seeing a live tab, and receiving meaningful updates during a visit. Once that core experience was clear, I could add loyalty, promotions, history, and admin analytics in a way that supported the main use case instead of distracting from it.

What I Learned About Technical Decisions

I learned that even a prototype benefits from deliberate architecture. Using JSON-backed storage kept the project lightweight, but pairing that with clear routes, reusable services, and real-time events made the system feel much more complete. I also saw how implementation details shape the portfolio story. Adding simulator scripts and checkout tooling did not only help development; those choices created stronger evidence for testing, iteration, and system behavior.

What I Learned About Communication

This capstone reinforced that good engineering work has to be explainable. Building the artifact package forced me to turn code structure into architecture diagrams, workflow diagrams, and annotations that show how the project changed over time. That process made the project easier to present because it highlighted why certain revisions mattered, not just what files changed.

How I Grew

I grew most in connecting implementation choices to user value. I did not just build screens or endpoints in isolation; I had to show how each part contributed to a more transparent dining experience. I also became more aware of how supporting tools, documentation, and repeatable demos affect the credibility of a project. By the end of the capstone, I was thinking more like a developer who has to design, build, test, explain, and defend a complete solution.
