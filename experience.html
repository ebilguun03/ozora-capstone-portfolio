Experience

The Capstone Experience

Ozora became more convincing once the project moved beyond interface ideas and started behaving like a complete software system. This page focuses on the four areas where that shift is most visible: architecture, ordering flow, data design, and revision workflow. Each theme keeps the early concept next to the implemented version so the project’s growth stays visible without turning the page into a long report.

Architecture

The earliest architecture only needed to prove that a customer could sign in, open a session, and read a live tab from a lightweight backend. As the implementation matured, the platform grew into a layered system with customer and operations surfaces, real-time delivery, shared backend services, and persistent records that support multiple features.

The first version centered on the smallest stack that could demonstrate a digital dining workflow.

UI and Ordering Workflow

The interface began as a short path into the dining session, with the live tab carrying most of the value. That early path evolved into a clearer workflow where sign-in, session creation, subscriptions, staged updates, refreshed totals, and checkout all connect to visible system state instead of isolated screens.

Early concept

The first product story emphasized getting the guest into the live tab quickly and treating the rest of the app as supporting expansion.

Data Model and Schema

The data model had to grow with the product. Early on, the core requirement was simply to represent a user, a dining session, and a tab. The implemented schema is broader because the system now carries that visit into receipts, loyalty, promotions, and menu-backed ordering records used across the customer and admin views.

Initial model

The first conceptual schema stayed intentionally small so the prototype could move quickly without overdesigning the domain.

erDiagram
    USER {
        string id
        string email
    }

    SESSION {
        string id
        string userId
    }

    TAB {
        string id
        string sessionId
    }

    LOYALTY_PROFILE {
        string userId
        int points
    }

    PROMOTION_FEED {
        string id
        string targetSegment
    }

    USER ||--o{ SESSION : starts
    SESSION ||--|| TAB : opens
    USER ||--|| LOYALTY_PROFILE : has
    USER }o--o{ PROMOTION_FEED : qualifies_for

Figure C1. Early data model focused on the minimum entities needed to support a personalized dining session.

Implemented schema

The final model keeps the dining visit at the center and then carries its results into longer-term customer records.

Testing and Revision

One of the most important technical improvements in the repository is not a screen or route but the revision workflow around them. The project moved from a mostly manual demo loop into a deterministic sequence that resets state, replays orders, simulates checkout, and verifies the downstream effects across history, loyalty, and admin analytics.

Early validation loop

At first, the project was validated by running the stack, sending an order, and checking the UI manually.

flowchart TD
    subgraph manual["Manual validation loop"]
        setup["Start backend, admin, and app"]
        simulate["Run order simulation"]
        observe["Observe tab changes"]
        adjust["Adjust code or seed data"]
    end

    setup --> simulate
    simulate --> observe
    observe --> adjust
    adjust --> setup

Figure D1. Early testing loop based on manual observation and repeated reruns.

Revised validation loop

The final workflow is stronger because it produces repeatable evidence instead of relying on one lucky demo run.

flowchart TD
    subgraph prep["Deterministic demo setup"]
        reset["Reset demo state"]
        start["Start session"]
    end

    subgraph replay["Replay and verify"]
        orders["Replay staged orders"]
        live["Verify live tab state"]
        close["Simulate checkout"]
    end

    subgraph evidence["Persisted outputs"]
        receipt["Verify receipt history"]
        loyalty["Verify loyalty changes"]
        admin["Verify admin analytics"]
    end

    reset --> start
    start --> orders
    orders --> live
    live --> close
    close --> receipt
    close --> loyalty
    close --> admin

Figure D2. Revised testing loop showing the repeatable workflow that made the demo and portfolio evidence much stronger.
