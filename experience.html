Experience

Technical evolution from prototype idea to connected dining system

Ozora became strongest when the project stopped reading like a collection of screens and started behaving like a system. The sections on this page focus on the four technical areas where that change is most visible: architecture, ordering flow, data design, and revision workflow. Each pair of artifacts keeps the early concept next to the implemented version so the project’s growth stays visible without turning the page into a long report.

Architecture

The earliest architecture only needed to prove that a customer could sign in, open a session, and read a live tab from a lightweight backend. As the project matured, the architecture expanded into a clearer platform model with multiple surfaces, real-time updates, shared backend logic, and repeatable demo tooling.

Early concept

The first version centered on the smallest stack that could demonstrate a digital dining workflow.

UI and Ordering Workflow

The UI started as a short path into the dining session, with the live tab carrying most of the product value. That early structure grew into a fuller workflow where sign-in, session creation, order updates, running totals, and checkout effects all connect to visible system state instead of isolated screens.

Early interface direction

The early product story focused on getting a guest into the dining session quickly and proving the tab concept first.

flowchart TD
    launch["Launch app"]
    login["Sign in"]
    home["Home"]
    tab["Live tab"]
    extras["Loyalty, offers, and history\nplanned after core flow"]

    launch --> login
    login --> home
    home --> tab
    tab --> extras

Figure B1. Early UI concept centered on a short path from sign-in to the live tab before the supporting customer features were fully developed.

Implemented ordering flow

The final workflow connects the interface to backend events, tab refreshes, and checkout side effects.

    session["Start session"]
    subscribe["Subscribe live tab"]
    update["Receive staged order update"]
    refresh["Refresh items and totals"]

    signin --> session
    session --> subscribe
    subscribe --> update
    update --> refresh
    refresh --> checkout

Figure B2. Implemented ordering flow showing how session state, live updates, and checkout now work as one connected experience.

Data Model and Schema

The data model had to grow with the product. Early on, the core requirement was simply to represent a user, a dining session, and a tab. The implemented schema is broader because the system now preserves receipts, loyalty changes, promotions, and menu structure across multiple parts of the experience.

Initial model

The first conceptual model stayed small enough to support a prototype without overdesigning the system.

flowchart TD
    user["User"]
    session["Session"]
    tab["Live tab"]
    profile["Loyalty and promotions"]

    user --> session
    session --> tab
    user --> profile

Figure C1. Early data model focused on the minimum relationships needed to support a personalized dining session.

Implemented schema

The final model keeps the dining visit as the center, then carries its results into longer-term customer records.

flowchart TD
    user["User profile"]
    visit["Session and tab"]
    receipt["Receipt history"]
    rewards["Loyalty and promotions"]
    menu["Menu catalog"]

    user --> visit
    menu --> visit
    visit --> receipt
    receipt --> rewards

Figure C2. Implemented schema showing how session data now feeds receipts, loyalty state, and menu-backed ordering records.

Testing and Revision

One of the most important technical improvements in the repository is not a screen or route but the revision workflow around them. The project moved from a mostly manual demo loop into a more deterministic sequence that resets state, replays orders, checks out a visit, and verifies the downstream effects across the customer and admin views.

Early validation loop

At first, the project was validated by starting the stack, triggering an order, and checking the UI manually.

flowchart TD
    setup["Start backend, admin, and app"]
    simulate["Run order simulation"]
    observe["Observe tab changes"]
    revise["Revise and rerun"]

    setup --> simulate
    simulate --> observe
    observe --> revise

Figure D1. Early testing loop based on manual observation and repeated reruns.

Revised validation loop

The final workflow is stronger because it produces repeatable evidence instead of relying on a single lucky demo run.

flowchart TD
    reset["Reset demo state"]
    start["Start session"]
    orders["Replay staged orders"]
    close["Simulate checkout"]
    verify["Verify receipt, loyalty, and admin analytics"]

    reset --> start
    start --> orders
    orders --> close
    close --> verify

Figure D2. Revised testing loop showing the repeatable workflow that made the final demo and portfolio evidence much stronger.
