Ozora Smart Dining Platform

Capstone Experience

This page documents how Ozora evolved from a focused mobile dining concept into a multi-surface smart dining platform with real-time tab updates, loyalty tracking, promotion management, and repeatable demo tooling. Each section pairs an early concept artifact with a revised implementation artifact so the portfolio shows technical growth over time instead of only the final state.

Theme 1

Architecture

Strong code support

Ozora strongly supports an architecture theme because the repository contains a SwiftUI client, an Express backend, Socket.io infrastructure, a separate admin dashboard, shared data services, and deterministic simulator scripts.

Figure A1. Early Concept Architecture for Ozora

The project started as a mobile-first restaurant prototype, so the earliest architecture was likely centered on proving that a customer app could log in, start a dining session, and read a live tab from a simple API. The current repository still shows that foundation in the separation between the iOS client, the Express routes, and the file-backed data service.

This artifact shows a focused first-pass architecture with a single client, a single backend, and JSON persistence. It is an early concept inferred from the present structure. What changed later was the addition of the admin dashboard, WebSocket-driven updates, and simulator tooling. That mattered because the project became a believable platform rather than a narrow mobile mockup. I learned that capstone architecture needs both user-facing flow and operational support.

Theme 2

UI and Ordering Workflow

Strong code support

This theme is strongly supported by the SwiftUI navigation structure, the five-tab interface, automatic session creation, and the staged order simulator that drives visible status changes in the Live Tab screen.

Figure B1. Early Mobile Information Architecture

The strongest user story in Ozora is the in-restaurant tab experience, so the earliest interface concept was likely organized around a short path from sign-in to a live order view. The final app still reflects that priority because the Live Tab remains the most behavior-rich screen in the client.

This artifact shows a likely early mobile IA focused on the primary customer task. It is an inferred early concept. What changed was that loyalty, promotions, and history became first-class tabs instead of remaining secondary ideas. That mattered because the experience evolved from a single-feature demo into a fuller customer journey. I learned that surrounding support screens make a flagship feature feel like part of a product instead of a lone demo.

Theme 3

Data Model and Schema

Strong code support

The repository strongly supports a schema theme because the backend data files define a real persistence model that is mirrored in the Swift `Codable` models and consumed by both the mobile app and the admin dashboard.

Figure C1. Early Conceptual Data Model

A restaurant platform can become over-modeled quickly, so the early data design needed to stay small enough to support a prototype. The core relationships between user, session, tab, loyalty account, and promotion represent the minimum data needed to prove personalized dining behavior.

This artifact shows the likely initial conceptual schema for the first working prototype. It is an early concept inferred from the implemented entity boundaries. What changed later was the addition of receipt history, menu structure, rewards, embedded order items, and richer lifecycle fields. That mattered because the system could now show continuity across visits. I learned that schema growth should follow user scenarios, especially when moving from one session to customer history.

Theme 4

Authentication and Access Boundaries

Moderate to strong code support

Authentication is not production-grade in Ozora, but it is still a meaningful capstone theme because the codebase includes a full mock OTP login path, persisted customer state, automatic session bootstrapping, and a clearly separate admin surface.

Figure D1. Early Access Assumption

Ozora is clearly a prototype, so the earliest access model did not need production-grade identity management. What mattered first was protecting the customer flow behind a sign-in step and keeping the staff workflow conceptually separate.

This artifact shows the early access assumption behind the project. It is an inferred early concept. What changed was the move to a concrete mock OTP flow, persistent local auth state, and automatic session setup after login. That mattered because the prototype started to feel like a real app with continuity between screens. I learned that even mocked authentication changes how seriously the rest of the experience is perceived.

Figure D2. Implemented Prototype Authentication and Access Flow

The implemented access flow is intentionally lightweight, but it is still specific enough to document as part of the portfolio. The repo shows a complete prototype path from sign-in to stored customer state, followed by automatic session creation and room-based real-time updates.

This artifact shows the real login and access boundaries present in the codebase. It is the revised version. The change was moving from a generic sign-in idea to a concrete mocked OTP workflow plus client-side persistence. That mattered because authentication, session state, and real-time subscriptions became connected. I learned that access flows influence system design far beyond the login screen.

Theme 5

Testing, Feedback, and Revision Evidence

Strong code support

This theme is supported by the README testing checklist and, more importantly, by real tooling in the repository that resets the demo state, simulates staged orders, closes a session, and exposes the results in both customer and admin views.

Figure E1. Early Manual Validation Loop

Early validation for this project was mainly experiential: start the stack, trigger an order, and visually confirm that the tab changed. That kind of manual loop is common in prototypes, and the README checklist still preserves that phase.

This artifact shows the likely original validation pattern before the demo tooling became more disciplined. It is an inferred early concept. What changed was the addition of reset and checkout scripts that make demos and verification more repeatable. That mattered because feedback could focus on behavior instead of setup luck. I learned that a strong capstone demo depends on the quality of the validation loop around it.

Figure E2. Revised Deterministic Demo and Validation Loop

The revised validation loop is one of the strongest hidden artifacts in the repository because it shows engineering maturity. Instead of only proving that orders can appear, the project now proves that an end-to-end visit can be reset, replayed, checked out, and reflected in history, loyalty, and admin analytics.

This artifact shows the implemented revision loop backed by actual npm scripts and downstream data updates. It is the revised version. Validation expanded from manual UI observation to a deterministic multi-step demo workflow. That mattered because it created stronger evidence for stakeholder feedback and presentation readiness. I learned that good demo tooling is part of the product story because it proves the system can survive repeated evaluation.
